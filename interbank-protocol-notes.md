# Banka 4 — Protocol Notes & Deviations

---

## Connection details

| Field | Value |
|---|---|
| Routing number | `444` |
| Interbank base URL | `http://rafsi.davidovic.io:8083` |
| API key (send in `X-Api-Key`) | `bank4-secret-key` |

### Interbank endpoints

All require `X-Api-Key: bank4-secret-key`.

| Method | Path | Description |
|---|---|---|
| `POST` | `/interbank` | 2PC payment (NEW_TX / COMMIT_TX / ROLLBACK_TX) |
| `GET` | `/interbank/public-stock` | List publicly offered stocks |
| `POST` | `/interbank/negotiations` | Create negotiation (buyer → seller's bank) |
| `GET` | `/interbank/negotiations/{rn}/{id}` | Get negotiation state |
| `PUT` | `/interbank/negotiations/{rn}/{id}` | Post counter-offer |
| `DELETE` | `/interbank/negotiations/{rn}/{id}` | Cancel negotiation |
| `GET` | `/interbank/negotiations/{rn}/{id}/accept` | Accept current offer |
| `GET` | `/interbank/user/{rn}/{id}` | Resolve our user to display name (`rn` must be `444`) |

`{rn}` is always the **seller's routing number**. `{id}` is the negotiation ID generated by the seller's bank on creation — both sides use the same `{rn}/{id}` pair.

## Deviations from the base protocol

| # | Area | Deviation | Impact on peer |
|---|---|---|---|
| 1 | `OtcOffer` | Extra field `buyerAccountNumber` (string) | **Required** — we reject offers without it (400) |
| 2 | Accept TX | 4-posting NEW_TX using OPTION asset + PERSON account | Peer must handle `"type":"OPTION"` asset and `"type":"PERSON"` account |
| 3 | Exercise TX | 4-posting NEW_TX using OPTION account type | Peer must handle `"type":"OPTION"` account |

## OTC Negotiation — OtcOffer

Sent on `POST /interbank/negotiations` (create) and `PUT /interbank/negotiations/{rn}/{id}` (counter-offer).

Routing number is sellers routing number and ID is same ID used on both sides to identify negotiation generated by seller on initial offer

### Extension: `buyerAccountNumber`

Not in the base spec. Agreed extension across all banks in this project.

The buyer must declare a specific account for settlement upfront. We use it to debit the premium on accept and the strike on exercise. Without it there is no way to know which account to charge at transaction time.

- Required on `POST` — missing = 400
- Immutable — cannot change on `PUT`, returns 400
- Immutable alongside: `stock.ticker`, `buyerId`, `sellerId`

### Full wire shape

```json
{
  "stock":              { "ticker": "CBSH" },
  "settlementDate":     "2027-06-01T00:00:00Z",
  "pricePerUnit":       { "currency": "USD", "amount": 44 },
  "premium":            { "currency": "USD", "amount": 2 },
  "buyerId":            { "routingNumber": 111, "id": "some-buyer-id" },
  "sellerId":           { "routingNumber": 444, "id": "10" },
  "amount":             10,
  "lastModifiedBy":     { "routingNumber": 111, "id": "some-buyer-id" },
  "buyerAccountNumber": "111000165540581321"
}
```

### Turn rule

Buyer's turn when `lastModifiedBy.routingNumber ≠ buyerId.routingNumber`. Acting out of turn returns 409.

Counter-offer `PUT` must include all fields (including immutable ones echoed back unchanged). Returns 204.

## Accept flow — NEW_TX

Triggered by `GET /interbank/negotiations/{sellerRn}/{id}/accept`. Moves the premium and forms the option contract.

**Sign:** `amount < 0` = asset leaves (credit out), `amount > 0` = asset enters (debit in).

| # | Account | Amount | Asset | Why this account type |
|---|---|---|---|---|
| 1 | `ACCOUNT` → `buyerAccountNumber` | `−premium` | `MONAS` | We use account specified by buyer in the offer |
| 2 | `PERSON` → `sellerId` | `+premium` | `MONAS` | Seller's bank will credit whichever account they choose for that identity |
| 3 | `PERSON` → `sellerId` | `−1` | `OPTION` | Seller gives the option right |
| 4 | `PERSON` → `buyerId` | `+1` | `OPTION` | Buyer receives the option right |
---

```json
{
  "idempotenceKey": { "routingNumber": 444, "locallyGeneratedKey": "peer-otc-accept-111-<negId>-new" },
  "messageType": "NEW_TX",
  "message": {
    "transactionId": { "routingNumber": 444, "id": "peer-otc-accept-111-<negId>" },
    "message": "Peer OTC option premium and contract acceptance",
    "paymentCode": "289",
    "paymentPurpose": "OTC option premium",
    "postings": [
      {
        "account": { "type": "ACCOUNT", "num": "111000165540581321" },
        "amount": -2,
        "asset": { "type": "MONAS", "asset": { "currency": "USD" } }
      },
      {
        "account": { "type": "PERSON", "id": { "routingNumber": 444, "id": "10" } },
        "amount": 2,
        "asset": { "type": "MONAS", "asset": { "currency": "USD" } }
      },
      {
        "account": { "type": "PERSON", "id": { "routingNumber": 444, "id": "10" } },
        "amount": -1,
        "asset": { "type": "OPTION", "asset": { "negotiationId": { "routingNumber": 444, "id": "<negId>" } } }
      },
      {
        "account": { "type": "PERSON", "id": { "routingNumber": 111, "id": "some-buyer-id" } },
        "amount": 1,
        "asset": { "type": "OPTION", "asset": { "negotiationId": { "routingNumber": 444, "id": "<negId>" } } }
      }
    ]
  }
}
```

**On COMMIT:** seller's bank creates a `PeerContract` and reserves the seller's shares.

## Exercise flow — NEW_TX

Triggered by `POST /api/peer-otc/contracts/{sellerRn}/{id}/exercise` from the buyer.

| # | Account | Amount | Asset | Why this account type |
|---|---|---|---|---|
| 1 | `ACCOUNT` → `buyerAccountNumber` | `−(price × qty)` | `MONAS` | Same settlement account declared in the offer |
| 2 | `OPTION` → `contractId` | `+(price × qty)` | `MONAS` | The seller's bank will find account connected with contract to credit to |
| 3 | `OPTION` → `contractId` | `−qty` | `STOCK` | Shares are taken from the contract, which holds the seller's share reservation |
| 4 | `PERSON` → `buyerId` | `+qty` | `STOCK` | Buyer receives the shares |

The `OPTION` account is a pseudo-account representing the contract itself. It differs from the OPTION asset in the accept flow:
- Accept: `OPTION` as **asset** — describes *what right* is being transferred
- Exercise: `OPTION` as **account** — the contract acts as escrow; funds and shares flow through it

```json
{
  "idempotenceKey": { "routingNumber": 444, "locallyGeneratedKey": "exec-<contractId>-new" },
  "messageType": "NEW_TX",
  "message": {
    "transactionId": { "routingNumber": 444, "id": "exec-<contractId>" },
    "message": "Peer OTC option exercise",
    "paymentCode": "289",
    "paymentPurpose": "OTC option exercise",
    "postings": [
      {
        "account": { "type": "ACCOUNT", "num": "444000112345678922" },
        "amount": -440,
        "asset": { "type": "MONAS", "asset": { "currency": "USD" } }
      },
      {
        "account": { "type": "OPTION", "id": { "routingNumber": 444, "id": "<contractId>" } },
        "amount": 440,
        "asset": { "type": "MONAS", "asset": { "currency": "USD" } }
      },
      {
        "account": { "type": "OPTION", "id": { "routingNumber": 444, "id": "<contractId>" } },
        "amount": -10,
        "asset": { "type": "STOCK", "asset": { "ticker": "CBSH" } }
      },
      {
        "account": { "type": "PERSON", "id": { "routingNumber": 444, "id": "10" } },
        "amount": 10,
        "asset": { "type": "STOCK", "asset": { "ticker": "CBSH" } }
      }
    ]
  }
}
```

> `strikePrice = 44`, `amount = 10` → `440 USD`.

**On COMMIT:** seller's bank marks the contract `exercised`, releases the share reservation, and credits the seller the strike.
